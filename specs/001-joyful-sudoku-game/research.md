## 1. Sudoku Puzzle Generation & Validation

### Algorithm: Backtracking with Unique Solution Verification

**Generation Strategy**:
```
1. Start with empty 9x9 grid
2. Fill grid using backtracking algorithm:
   - Place random valid numbers (1-9) in cells
   - Backtrack when no valid number can be placed
   - Continue until grid is completely filled
3. Remove numbers strategically based on difficulty:
   - Easy: Remove 30-35 numbers (46-51 given)
   - Medium: Remove 40-45 numbers (36-41 given)
   - Hard: Remove 50-55 numbers (26-31 given)
4. Verify puzzle has exactly one solution using solver
5. If multiple solutions exist, remove fewer numbers and retry
```

**Key Insights**:
- **Symmetry**: Removing numbers symmetrically (e.g., opposite corners) improves perceived puzzle quality
- **Performance**: Pre-generating puzzles in Web Worker prevents UI blocking (~50-200ms per puzzle)
- **Caching**: Store 2-3 pre-generated puzzles per difficulty in IndexedDB for instant "New Game" response

**Validation Algorithm** (Real-time conflict detection):
```typescript
function validateMove(grid: number[][], row: number, col: number, value: number): boolean {
  // Check row for duplicates
  for (let c = 0; c < 9; c++) {
    if (c !== col && grid[row][c] === value) return false;
  }

  // Check column for duplicates
  for (let r = 0; r < 9; r++) {
    if (r !== row && grid[r][col] === value) return false;
  }

  // Check 3x3 box for duplicates
  const boxStartRow = Math.floor(row / 3) * 3;
  const boxStartCol = Math.floor(col / 3) * 3;
  for (let r = boxStartRow; r < boxStartRow + 3; r++) {
    for (let c = boxStartCol; c < boxStartCol + 3; c++) {
      if ((r !== row || c !== col) && grid[r][c] === value) return false;
    }
  }

  return true;
}
```

**Reference Libraries** (for inspiration, not direct use):
- `sudoku` npm package: Good reference for generation algorithm
- Custom implementation preferred for learning and full control

## 2. PWA Architecture & Offline Functionality

### Service Worker Strategy: Cache-First with Network Fallback

**Tools**:
- **vite-plugin-pwa**: Automatic service worker generation with Workbox
- **Workbox**: Google's production-ready service worker library

**Caching Strategy**:
```javascript
// Static Assets (HTML, CSS, JS, fonts, icons): Precache on install
// Generated by vite-plugin-pwa automatically

// Game State: IndexedDB (not cached by service worker)
// User-generated data persisted via Dexie.js
```

**Configuration** (`vite.config.ts`):
```typescript
import { VitePWA } from 'vite-plugin-pwa'

export default defineConfig({
  plugins: [
    VitePWA({
      registerType: 'autoUpdate',
      manifest: {
        name: 'Joyful Sudoku',
        short_name: 'Sudoku',
        description: 'A delightful puzzle game designed with love',
        theme_color: '#f3e5f5', // Soft pastel purple
        background_color: '#ffffff',
        display: 'standalone', // Hides browser UI
        icons: [
          { src: 'icon-192.png', sizes: '192x192', type: 'image/png' },
          { src: 'icon-512.png', sizes: '512x512', type: 'image/png' }
        ]
      },
      workbox: {
        globPatterns: ['**/*.{js,css,html,ico,png,svg,woff2}'],
        runtimeCaching: [] // No network requests for core gameplay
      }
    })
  ]
})
```

**State Persistence**: IndexedDB via Dexie.js

**Why IndexedDB over localStorage**:
- ✅ Structured data storage (no JSON.stringify overhead)
- ✅ Handles large datasets efficiently (undo history with 100+ moves)
- ✅ Transactional safety (atomic saves prevent corruption)
- ✅ Asynchronous API (non-blocking)

**Schema Design**:
```typescript
// Dexie.js table definition
class SudokuDatabase extends Dexie {
  gameState!: Table<GameState, number>;
  progress!: Table<Progress, string>;

  constructor() {
    super('SudokuDB');
    this.version(1).stores({
      gameState: '++id, timestamp',  // Auto-increment ID, indexed timestamp
      progress: 'difficulty'          // Key by difficulty level
    });
  }
}
```

## 3. Animation Performance & Delightful Micro-Interactions

### Framework: Framer Motion

**Why Framer Motion**:
- ✅ Hardware-accelerated CSS transforms (60fps on mobile)
- ✅ Declarative API integrates seamlessly with React
- ✅ Built-in gesture support (tap, drag, hover)
- ✅ Layout animations automatically smooth transitions
- ✅ Production-ready with excellent TypeScript support

**Animation Patterns**:

**1. Number Entry (< 100ms)**:
```tsx
<motion.div
  initial={{ scale: 0.8, opacity: 0 }}
  animate={{ scale: 1, opacity: 1 }}
  transition={{ duration: 0.15, ease: 'easeOut' }}
>
  {number}
</motion.div>
```

**2. Row/Box Completion**:
```tsx
<motion.div
  animate={{ backgroundColor: ['#f3e5f5', '#e1bee7', '#f3e5f5'] }}
  transition={{ duration: 0.6, ease: 'easeInOut' }}
/>
```

**3. Puzzle Completion Celebration**:
```tsx
import confetti from 'canvas-confetti';

// Trigger on puzzle complete
confetti({
  particleCount: 100,
  spread: 70,
  origin: { y: 0.6 },
  colors: ['#f3e5f5', '#e1bee7', '#ce93d8']
});
```

**Performance Optimization**:
- Use `transform` and `opacity` only (GPU-accelerated properties)
- Avoid animating `width`, `height`, `top`, `left` (triggers layout reflow)
- Use `will-change: transform` sparingly (pre-optimization)
- Lazy load celebration animation components (reduce initial bundle)

**Alternative Library**: React Spring (physics-based animations)
- **Decision**: Framer Motion chosen for simpler API and better documentation

## 4. Testing Strategy

### Test Pyramid for Sudoku PWA

**Layer 1: Unit Tests (Vitest)** - 60% of tests
- **Coverage**: Game logic (`lib/puzzle/`, `lib/state/`)
- **Tools**: Vitest (Vite-native, fast), `@testing-library/react` for component logic
- **Examples**:
  - Puzzle generation produces valid grids
  - Validator correctly detects conflicts
  - Solver finds unique solution
  - State actions (undo, enterNumber) work correctly

```typescript
// Example: generator.test.ts
import { describe, it, expect } from 'vitest';
import { generatePuzzle } from './generator';
import { solvePuzzle } from './solver';

describe('Puzzle Generator', () => {
  it('generates puzzle with unique solution', () => {
    const puzzle = generatePuzzle('medium');
    const solutions = solvePuzzle(puzzle);
    expect(solutions).toHaveLength(1);
  });

  it('respects difficulty level for given count', () => {
    const easyPuzzle = generatePuzzle('easy');
    const givenCount = easyPuzzle.flat().filter(n => n !== 0).length;
    expect(givenCount).toBeGreaterThanOrEqual(40);
    expect(givenCount).toBeLessThanOrEqual(50);
  });
});
```

**Layer 2: Integration Tests (React Testing Library)** - 30% of tests
- **Coverage**: Component interactions, state updates, user flows
- **Examples**:
  - Selecting cell shows number pad
  - Entering number updates grid and validates
  - Undo/Redo maintains state consistency
  - Completing puzzle triggers celebration

```typescript
// Example: game-flow.test.tsx
import { render, screen, fireEvent } from '@testing-library/react';
import { App } from '../App';

test('completing puzzle shows celebration', async () => {
  render(<App />);

  // Select easy difficulty
  fireEvent.click(screen.getByText('Easy'));

  // Fill in remaining cells with correct answers
  // ... (test helper to fill valid solution)

  // Assert celebration animation appears

```

**Layer 3: E2E Tests (Playwright)** - 10% of tests
- **Coverage**: Critical user journeys, PWA functionality, cross-browser
- **Examples**:
  - Complete first puzzle (P1 user story)
  - Use stress-free tools (P2 user story)
  - App works offline after initial visit
  - State persists across app close/reopen

```typescript

  await page.goto('/');
```

## 5. Deployment Strategy

### Platform: Vercel (Recommended)

**Why Vercel over Netlify**:
- ✅ Zero-config for Vite projects
- ✅ Automatic HTTPS and global CDN
- ✅ Preview deployments for every git push
- ✅ Analytics and performance monitoring included
- ✅ Edge network optimized for low latency

**Deployment Flow**:
1. Connect GitHub repository to Vercel
2. Configure build command: `npm run build`
3. Set output directory: `dist`
4. Push to main branch → Auto-deploy to production
5. Push to feature branch → Preview URL generated

**Custom Domain** (optional):
```
joyful-sudoku.vercel.app (default)
sudoku.yourdomain.com (custom, requires DNS configuration)
```

**Environment Variables**: None required (fully client-side app)

## Key Decisions Summary

| Decision | Choice | Rationale |
|----------|--------|-----------|
| **Build Tool** | Vite | 10-100x faster than CRA, better DX, native TypeScript/PWA support |
| **State Management** | Zustand | Simpler than Redux, 1KB size, excellent TypeScript support |
| **Storage** | IndexedDB (Dexie.js) | Structured data, transactional safety, handles large undo history |
| **Animation** | Framer Motion | Hardware-accelerated, declarative API, production-ready |
| **Testing** | Vitest + Playwright | Fast unit tests, comprehensive E2E coverage, Vite-native |
| **Deployment** | Vercel | Zero-config, auto-deploy, global CDN, preview URLs |
